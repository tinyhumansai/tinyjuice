# Adapter And Product Surfaces Plan

## Goal

Add adapter-facing surfaces only after the core protocol, safety policy, and
storage contracts are stable.

## Artifact Store

CCR is for model-facing recovery of compacted originals. An artifact store is
for operator inspection, stats, discovery, and optional raw storage.

Plan:

- Add artifact ids generated by TinyJuice, not by user paths or URLs.
- Use private file permissions where supported.
- Support metadata-only records when raw storage is disabled.
- Track source, command, exit code, raw chars, reduced chars, ratio,
  classification, creation time, and truncation state.
- Keep CCR tokens and artifact ids separate.

Acceptance:

- Invalid artifact ids cannot escape the artifact directory.
- Stats can be computed without storing raw content.
- Raw artifact storage is opt-in.

## Web Extract Reducer

Plan:

- Accept already extracted markdown/text/HTML plus URL metadata.
- Do not fetch URLs.
- Strip inline base64 image payloads.
- Return small pages whole after image conversion.
- Store large pages and return line-snapped head/tail with retrieval footer.
- Apply combined budget for multi-page batches without dropping footers.

Acceptance:

- Real image URLs remain.
- Base64 image bytes are absent from output and metadata.
- Retrieval handle is emitted only when storage succeeds.
- Full URLs with tokens are not logged.

## SQL Context Reducer

Plan:

- Define schema IR types in core.
- Let adapters provide parsed migration facts or live introspection results.
- Rank relevant tables, columns, indexes, and foreign-key edges.
- Render compact row sets with capped rows and cell widths.
- Report safe dialect rewrites.

Acceptance:

- Mutating SQL is rejected in reducer paths.
- Credentials are never stored or reported.
- Live data and migration-derived facts are labeled separately.

## Subagent Summary Surface

Plan:

- Define transcript/event types and evidence reference types.
- Add deterministic evidence extraction for paths, line numbers, tool names, and
  final conclusions.
- Leave model-based summarization to host adapters.
- Return findings, evidence, uncertainty, and omitted counts.

Acceptance:

- Summary does not invent evidence.
- Contradictory findings remain as uncertainty.
- Byte budget produces an omission report.

## Batched Edit And Validation Surface

Plan:

- Keep mutation in host adapters.
- Core can define edit reports, validation result types, and compact validation
  output helpers.
- Support exact and carefully bounded fuzzy match reporting.

Acceptance:

- Ambiguous fuzzy matches are rejected.
- Validation failures return compact actionable evidence.
- Full post-edit files are not echoed by default.

## Host Installers And Doctor

Plan:

- Implement only after CLI and protocol are stable.
- Add a host registry for install, uninstall, doctor, and runtime hook command.
- Start with OpenHuman and Codex.
- Keep host-specific mutation code out of core compression modules.

Acceptance:

- Install is idempotent.
- Uninstall is idempotent.
- Doctor reports `ok`, `warn`, `broken`, or `disabled`.
- Unrelated config keys are preserved.

## Savings And Observability

Plan:

- Expand savings records into counted, measured, and estimated classes.
- Record applied compressor, content kind, byte counts, token estimates, CCR
  state, rule id, and skip reasons.
- Add benchmark fixture reports separately from live stats.

Acceptance:

- No raw content in serialized accounting records.
- Cost savings labels distinguish measured from estimated.
- UI can explain fallback frequency by command family.
